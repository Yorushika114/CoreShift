# CLAUDE.md

## 项目目标

围绕题目一「语音版的日历工具」，开发一个以语音为核心入口的日历管理工具。

这个产品应该帮助用户通过自然语音更高效、更便捷地管理日程事件。核心体验不是“普通日历界面加一个麦克风按钮”，而是让用户可以自然地说话，并完成添加、删除、查看、提醒管理等完整日历操作。

## 题目要求

实现一个以语音交互为核心的日历管理工具。

必须支持的能力：

- 通过语音添加日历事件。
- 通过语音删除日历事件。
- 通过语音查看日历事件。
- 通过语音创建和管理事件提醒。
- 体现对用户在日历管理中真实需求的理解。
- 当语音识别或语义解析不确定时，能够准确、顺畅地追问和恢复。

项目需要在活动周期内完成一个可演示、可运行的 MVP。优先打磨核心流程，不要堆很多浅层功能。

## 产品方向

围绕真实日历管理痛点设计：

- 用户不想手动填写冗长表单。
- 用户说出的时间表达可能不完整或有歧义。
- 删除等危险操作必须二次确认。
- 用户需要快速得到答案，例如“我明天下午有什么安排？”
- 用户需要能轻松创建和检查提醒。

优先支持这类自然对话：

- “明天下午三点提醒我开组会”
- “下周一上午九点到十点安排算法课”
- “删除明天三点的组会”
- “我今天还有什么安排”
- “把今晚八点的复习提醒改到九点”

## 核心体验原则

- 语音优先：可以提供文字输入作为兜底，但主流程应能通过语音完成。
- 重要操作要确认：尤其是删除、歧义创建、批量修改等操作。
- 过程透明：展示系统听到了什么、解析出了什么、准备执行什么。
- 优雅恢复：如果时间、标题、目标事件不明确，要追问澄清问题。
- 日历可读：用户始终可以通过视觉界面检查未来事件。
- 避免伪装能力：如果使用浏览器语音识别、外部语音 API 或模拟语音输入，需要在 README 中明确说明。

## 建议 MVP 范围

优先实现：

- 语音输入采集，或模拟语音输入。
- 面向中文日期/时间表达的自然语言事件解析器。
- 创建事件：包含标题、开始时间、可选结束时间、提醒时间。
- 按日期分组展示事件列表。
- 支持用语音查询今天、明天、本周或指定日期的事件。
- 支持带确认的语音删除。
- 使用浏览器存储或简单后端数据库进行本地持久化。

可选增强：

- 编辑或改期事件。
- 重复事件。
- 提醒通知。
- 日程冲突检测。
- 多轮澄清。
- 日历数据导入/导出。

## 架构建议

保持架构简单、清晰、可解释。

推荐模块：

- `voice`：语音识别、录音状态、转写文本处理，或模拟语音输入。
- `nlp`：意图识别与中文时间解析。
- `calendar`：事件模型、增删改查、冲突检查、提醒逻辑。
- `ui`：日历视图、事件列表、确认弹窗、语音反馈。
- `storage`：localStorage、IndexedDB、SQLite 或 API 持久化。

如果做纯前端 MVP，优先选择结构清楚的单页应用，并使用本地持久化。如果使用后端，请将项目拆成 `/frontend` 和 `/backend` 目录。

## 数据模型

使用明确的事件模型：

```ts
type CalendarEvent = {
  id: string;
  title: string;
  startAt: string;
  endAt?: string;
  reminderAt?: string;
  createdAt: string;
  updatedAt: string;
  sourceText?: string;
};
```

内部时间统一使用 ISO 格式存储。展示时使用符合中文语境的本地化格式。

## 语音与 NLP 行为

每次拿到语音转写文本后：

1. 识别意图：创建、删除、查询、修改、未知。
2. 提取事件标题。
3. 提取日期、时间和持续时间。
4. 检查是否存在歧义。
5. 要求用户确认或补充信息。
6. 执行日历操作。
7. 展示简洁明确的结果反馈。

不确定时不要静默猜测，要主动追问。

示例：

- 用户说“明天提醒我开会”，但没有时间，则追问“明天几点提醒你？”
- 用户说“删除明天的会”，但匹配到多个事件，则展示候选项并询问删除哪一个。
- 用户只说“下午三点”，但缺少日期；如果该时间尚未过去，可以默认今天，否则应追问日期。

## 实现规则

- 不要只硬编码演示语句。解析能力可以有限，但要能处理常见表达变体。
- 依赖保持轻量，并在 README 中列出所有第三方依赖。
- 源码应可读、模块化、便于评审。
- 不要提交生成缓存、大型构建产物或本地环境文件。
- 只在非显而易见的解析逻辑或调度逻辑处添加注释。
- UI 状态要清晰，包括空状态、加载中、监听中、解析中、待确认、成功、错误等。

## 测试要求

为以下能力添加聚焦测试：

- 意图识别。
- 中文时间解析。
- 事件创建。
- 删除事件时的匹配逻辑。
- 按日期范围查询过滤。
- 歧义命令处理。

如果时间紧张，至少在 README 中提供手动测试清单，覆盖必要的语音流程。

## README 要求

README 必须包含：

- 项目概述和所选题目。
- 核心功能。
- 技术栈和依赖。
- 哪些部分是原创实现。
- 安装和运行方式。

- Demo 视频链接。

## PR 与 Commit 规范

活动规则重视开发过程质量、PR 质量和 commit 分布。

使用小 PR。每个 PR 只做一件事，例如：

- 项目脚手架。
- 语音输入 UI。
- NLP 命令解析器。
- 日历事件模型与存储。
- 创建事件流程。
- 查询事件流程。
- 删除事件流程。
- 提醒流程。
- README 和 demo 打磨。

每个 PR 描述应包含：

- 改了什么。
- 为什么重要。
- 实现思路。
- 如何测试。

每次合并后都要保证 `main` 分支可运行。

## Demo 演示脚本

准备一个演示，至少展示：

1. 用户说出创建事件的语音命令。
2. 系统展示转写文本和解析出的事件信息。
3. 用户确认创建。
4. 事件出现在日历中。
5. 用户询问今天或明天有什么安排。
6. 系统回答并筛选日历事件。
7. 用户通过语音删除事件。
8. 系统要求确认，然后删除事件。
9. 展示提醒行为或提醒配置。

Demo 应在前一分钟内让评审看懂“语音优先”的价值。

## 完成标准

满足以下条件时，项目可以提交：

- 必要的添加、删除、查看、提醒流程可用。
- 从干净 clone 后能按照 README 跑起来。
- UI 清楚展示语音转写、解析意图和执行结果。
- 有歧义或风险的命令需要澄清或确认。
- README 包含依赖、原创性、运行方式和 demo 信息。
- Demo 视频链接可访问。
- Git 历史体现持续提交和有意义的 PR。
